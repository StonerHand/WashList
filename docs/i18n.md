# WashList i18n Guide

## Supported Locales

| Surface | Locales | Notes |
| --- | --- | --- |
| Landing | `en`, `ru`, `de`, `es`, `fr` | Public marketing copy. |
| App | `en`, `ru` | Full product UI dictionaries. Unsupported stored languages safely fall back to English. |

Landing and app keep local dictionaries in JavaScript because the project is static and has no build step. If more app locales are added, move dictionaries into separate JSON files and lazy-load them per language.

## Key Rules

- Use stable semantic keys, not text-based keys.
- Keep button labels short.
- Keep error messages human: what happened and what to do next.
- Reuse the same product terms everywhere.
- Use `Intl.DateTimeFormat` and `Intl.NumberFormat` for dates and numbers.
- Do not show raw keys. Fallback to English.

## Translation Workflow

Recommended process for production:

1. Draft UI strings in English.
2. Translate with DeepL as a first pass.
3. Adapt tone and context with ChatGPT or a product copywriter.
4. Manage source strings in Crowdin, Lokalise, or Phrase.
5. Use glossary and translation memory for repeated product terms.
6. Review key markets with a native speaker.
7. Attach screenshots to translation tasks so labels are reviewed in context.

## QA Checklist

- Switch every supported locale on landing and app.
- Verify no raw keys are visible.
- Verify long German/French landing strings do not overflow buttons, cards, or nav.
- Verify Russian wording stays short and natural.
- Verify language choice persists after reload.
- Verify signed-out auth gate strings are localized on every app route.
- Verify connected-but-load-failed state says the profile is connected and offers retry/sign-out actions.
- Verify compare rows keep artwork, title, artist, duration, and shared state readable in long translations.
- Verify dates and numbers are locale-formatted.
- Verify HTML-enabled marketing strings are sanitized to the allowed tags.
